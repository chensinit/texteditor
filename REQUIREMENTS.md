# Text Editor & Viewer Requirements

## 1. Product Goal

macOS에서 동작하는 탭 기반 텍스트 에디터/뷰어를 만든다.

핵심 방향:

- 검정 계열 배경의 집중형 편집 화면
- 일반 텍스트를 빠르고 안정적으로 입력/열람 가능
- 긴 줄은 가로 스크롤 없이 다음 줄로 자연스럽게 표시
- 여러 문서를 탭으로 동시에 열고 관리
- 저장하지 않은 신규 문서는 앱이 종료되거나 탭이 닫히기 전까지 임시 데이터로 보존
- “가볍지만 부족하지 않은” 기본 편의 기능 제공

이 문서는 MVP 범위와 이후 확장 가능한 요구사항을 함께 정의한다.

## 2. Target Users

- 메모를 빠르게 작성하는 사용자
- 로그, 설정 파일, 일반 텍스트를 확인하는 사용자
- 복잡한 IDE 대신 단순하고 빠른 편집기를 원하는 사용자

## 3. Core Principles

- 실행 즉시 바로 입력 가능해야 한다.
- 텍스트 편집의 기본기는 빠지면 안 된다.
- 저장 여부와 문서 상태가 명확하게 보여야 한다.
- UI는 단순하되, 실제 사용 시 답답하지 않아야 한다.
- 대용량 텍스트에서도 가능한 한 부드럽게 동작해야 한다.

## 4. Main Use Cases

### 4.1 신규 문서 작성

- 사용자는 새 탭을 열어 즉시 텍스트를 입력할 수 있어야 한다.
- 저장 전까지 문서는 임시 문서 상태로 관리된다.
- 앱이 비정상 종료되지 않는 한, 임시 문서는 복구 가능해야 한다.

### 4.2 기존 파일 열기

- 사용자는 로컬 텍스트 파일을 열어 내용을 볼 수 있어야 한다.
- 연 파일은 읽기/쓰기 가능 상태로 열리는 것을 기본으로 한다.
- 인코딩 문제나 읽기 실패 시 원인을 사용자에게 알려야 한다.

### 4.3 편집 후 저장

- 사용자는 현재 문서를 저장하거나 다른 이름으로 저장할 수 있어야 한다.
- 저장되지 않은 변경사항은 명확히 표시되어야 한다.

### 4.4 다중 탭 작업

- 사용자는 여러 문서를 탭으로 열고 전환할 수 있어야 한다.
- 각 탭은 제목, 저장 상태, 파일 경로 여부를 구분해서 보여야 한다.

### 4.5 읽기 중심 사용

- 사용자는 긴 텍스트를 줄번호와 함께 안정적으로 읽을 수 있어야 한다.
- 텍스트가 오른쪽으로 길어지면 자동 줄바꿈으로 이어서 보여야 한다.

## 5. Functional Requirements

## 5.1 Editor Layout

- 기본 편집 영역 배경은 검정 또는 매우 어두운 회색 계열이어야 한다.
- 기본 텍스트 색상은 높은 대비를 유지해야 한다.
- 편집 화면은 최소 다음 영역으로 구성한다.
- 상단: 툴바
- 툴바 하단 또는 동일 상단 영역: 탭 바 또는 문서 전환 영역
- 좌측: 줄번호 영역
- 중앙: 텍스트 편집 영역
- 필요 시 하단: 상태 바
- 툴바와 편집 영역은 시각적으로 분리되어야 하며, 문서 중심 화면을 해치지 않아야 한다.

## 5.1.1 Top Toolbar

- 상단 툴바를 기본 UI 요소로 둔다.
- 툴바는 자주 사용하는 기능에 빠르게 접근하기 위한 목적이어야 한다.
- 툴바는 아이콘만으로 끝내지 말고, 필요한 항목은 라벨 또는 툴팁을 제공해야 한다.
- 창 폭이 줄어들면 우선순위가 낮은 항목은 overflow 또는 메뉴로 이동 가능해야 한다.
- 툴바 구성은 최소 다음 그룹을 포함한다.

파일 그룹:
- 새 문서
- 파일 열기
- 저장
- 다른 이름으로 저장

편집 그룹:
- 실행 취소
- 다시 실행
- 잘라내기
- 복사
- 붙여넣기

서식/표시 그룹:
- 폰트 선택
- 글자 크기 증가
- 글자 크기 감소
- 줄바꿈 on/off 토글 또는 표시 상태
- 탭 폭 또는 들여쓰기 설정 진입점

탐색 그룹:
- 검색 열기
- 줄 이동(Go To Line) 진입점

보기/보조 그룹:
- 현재 줄 강조 on/off
- 공백/탭/줄바꿈 기호 표시 on/off
- 테마 선택 진입점
- 배경 이미지 on/off 또는 설정 진입점

- 툴바에는 현재 문서 제목 또는 경로 일부를 표시하는 방안도 검토한다.
- 저장되지 않은 문서인 경우 툴바에서도 상태를 확인할 수 있으면 좋다.
- 툴바 버튼은 단축키가 있는 경우 동일 기능과 일관되게 동작해야 한다.

## 5.2 Text Input and Editing

- 일반 문자열 입력
- 한글/영문/숫자/기호 입력
- 붙여넣기, 잘라내기, 복사
- 실행 취소/다시 실행
- 전체 선택
- 드래그 선택 및 키보드 선택
- 마우스 클릭으로 커서 이동
- 다중 줄 선택 지원

## 5.3 Tab Support

- 새 탭 생성
- 탭 닫기
- 탭 간 전환
- 저장되지 않은 문서가 있는 탭 닫기 시 확인 대화상자 표시
- 탭 제목 규칙
- 저장 전 신규 문서: `Untitled`, `Untitled 2` 등
- 저장 후 문서: 파일명 표시
- 변경사항이 있으면 탭 제목 또는 아이콘으로 수정 상태 표시

## 5.4 Line Numbers

- 모든 줄에 대해 좌측 줄번호 표시
- 현재 커서가 위치한 줄은 줄번호 또는 배경을 강조
- 줄번호 영역은 편집 영역과 시각적으로 구분
- 줄 수가 늘어나면 번호 영역 폭이 자동 조정

## 5.5 Wrapping and Scrolling

- 기본 동작은 소프트 랩(soft wrap)이다.
- 긴 줄은 같은 논리 줄 내에서 다음 표시 줄로 이어진다.
- 줄번호는 논리 줄 기준으로 유지한다.
- 기본적으로 가로 스크롤은 사용하지 않는다.
- 세로 스크롤은 지원한다.
- 추후 옵션으로 “줄바꿈 끄기”를 제공할 수 있도록 구조를 설계한다.

## 5.6 Tabs and Indentation

- Tab 키 입력 지원
- Shift+Tab 들여쓰기 감소 지원
- 여러 줄 선택 상태에서 Tab/Shift+Tab 일괄 들여쓰기 지원
- 탭 정책은 최소 다음 중 하나를 지원해야 한다.
- 실제 탭 문자 삽입
- 공백으로 변환 삽입
- 기본값은 요구사항 결정 후 선택하되, 내부 구조는 추후 설정 변경 가능해야 한다.

## 5.7 File Operations

- 새 문서
- 파일 열기
- 저장
- 다른 이름으로 저장
- 최근 연 파일 목록
- 파일 경로가 있는 문서의 경우 빠른 저장 지원
- 외부에서 파일이 변경되었을 때 재로드 또는 충돌 처리 정책 필요

## 5.7.1 File Format Detection

- 앱은 파일을 열 때 텍스트 형식을 가능한 범위 내에서 자동 판단해야 한다.
- 자동 판단 대상은 최소 다음을 포함한다.
- 인코딩 추정(예: UTF-8, UTF-16 등)
- 줄바꿈 형식 추정(LF, CRLF)
- 들여쓰기 스타일 추정(탭 기반 / 공백 기반)
- 읽기 전용 가능 여부 또는 쓰기 권한 상태
- 파일 확장자 기반의 문서 종류 추정(plain text, markdown, json, xml, source code 등)
- 추정 결과는 상태 바 또는 문서 정보 UI에서 확인 가능해야 한다.
- 자동 판단 결과가 틀릴 수 있으므로 사용자가 수동으로 변경할 수 있어야 한다.
- 바이너리 파일로 보이는 입력은 일반 텍스트로 무리하게 열지 말고 경고 또는 읽기 제한 처리해야 한다.
- MVP에서는 문법 하이라이팅까지 포함하지 않더라도, 문서 종류 추정 값은 이후 확장 가능하도록 유지한다.

## 5.8 Temporary Draft Persistence

- 저장하지 않은 신규 탭 문서는 임시 데이터로 자동 저장한다.
- 자동 저장 시점
- 일정 시간 간격
- 텍스트 변경 후 debounce
- 앱이 백그라운드로 갈 때
- 앱 종료 시
- 임시 저장 데이터에는 최소 다음 정보가 포함된다.
- 임시 문서 ID
- 탭 제목
- 본문 텍스트
- 커서/선택 범위
- 마지막 수정 시각
- 저장된 파일과 연결되지 않은 문서는 앱 재실행 시 복원 가능해야 한다.
- 사용자가 명시적으로 닫은 임시 문서는 복원 목록에서 제거한다.

## 5.9 Session Restore

- 앱 재실행 시 이전 세션의 열린 탭을 복원할 수 있어야 한다.
- 복원 대상
- 저장된 파일 탭
- 저장되지 않은 임시 탭
- 복원 시 마지막 활성 탭과 스크롤 위치를 가능한 범위 내에서 유지한다.

## 5.10 Search

- 현재 문서 내 문자열 검색
- 다음 결과 / 이전 결과 이동
- 검색 결과 수 표시
- 대소문자 구분 옵션
- 정규식은 MVP에서는 제외 가능

## 5.11 Replace

- 현재 문서 내 문자열 치환
- 현재 항목 치환
- 전체 치환
- 치환 전 검색 결과를 기반으로 동작해야 한다.

## 5.12 Status Bar

- 상태 바에는 최소 다음 정보를 표시한다.
- 현재 줄, 현재 열
- 선택된 문자 수 또는 선택 범위
- 인코딩 정보
- 감지된 문서 형식 또는 파일 타입 정보
- 줄바꿈 형식(LF/CRLF) 확장 가능 영역
- 저장 상태
- 읽기 전용 여부
- 현재 폰트 크기 또는 보기 배율 확장 가능 영역

## 5.13 Keyboard Shortcuts

- `Cmd+N`: 새 탭
- `Cmd+O`: 파일 열기
- `Cmd+S`: 저장
- `Shift+Cmd+S`: 다른 이름으로 저장
- `Cmd+W`: 탭 닫기
- `Cmd+F`: 검색
- `Cmd+Z`: 실행 취소
- `Shift+Cmd+Z`: 다시 실행
- `Cmd+A`: 전체 선택
- `Cmd+Tab`과 충돌하는 앱 전역 단축키는 피한다.

## 5.14 Drag and Drop

- text 파일을 앱 창으로 드래그하면 새 탭으로 열 수 있어야 한다.
- 여러 파일 드롭 시 여러 탭으로 열 수 있어야 한다.
- 드래그된 파일이 텍스트로 해석 가능한지 검사해야 한다.
- 지원하지 않는 형식 또는 바이너리 파일은 경고 후 열지 않거나 읽기 제한 상태로 처리해야 한다.

## 5.15 Read-Only Handling

- 읽기 전용 파일은 읽기 전용 상태로 열 수 있어야 한다.
- 편집 불가 이유를 사용자에게 명확히 보여야 한다.
- 다른 이름으로 저장은 허용한다.

## 6. Non-Functional Requirements

## 6.1 Performance

- 일반적인 메모/텍스트 편집은 즉시 반응해야 한다.
- 최소 수만 줄 규모의 텍스트에서도 스크롤과 입력 지연이 과도하지 않아야 한다.
- 줄번호 계산과 줄바꿈 처리 때문에 입력 지연이 커지지 않도록 한다.
- 대용량 파일은 일반 모드와 다른 로딩 전략을 사용할 수 있어야 한다.
- 대용량 파일 오픈 시 앱 전체가 멈춘 것처럼 보이지 않아야 한다.

## 6.2 Reliability

- 앱 강제 종료 또는 비정상 종료에 대비해 임시 문서 복구가 가능해야 한다.
- 자동 저장은 UI를 멈추게 하지 않아야 한다.
- 저장 실패 시 사용자 데이터 손실이 없어야 한다.

## 6.3 Usability

- 저장되지 않은 상태 표시가 명확해야 한다.
- 파일이 실제 저장된 문서인지 임시 문서인지 혼동되지 않아야 한다.
- 기본 테마만으로도 장시간 읽기/입력에 무리가 없어야 한다.

## 6.4 Accessibility

- 시스템 기본 접근성 기능과 충돌하지 않아야 한다.
- 폰트 크기 확대에 대응해야 한다.
- 충분한 색상 대비를 유지해야 한다.

## 7. Recommended Additional Features

아래 항목은 사용자가 처음 언급하지 않았지만, 실제 완성도에 큰 차이를 만드는 기능이다.

### 7.1 Current Line Highlight

- 현재 커서 줄 배경 강조
- 긴 문서에서 시선 추적이 쉬워진다.

### 7.2 Invisible Characters Toggle

- 공백, 탭, 줄바꿈 기호 표시 옵션
- 들여쓰기 확인과 텍스트 정리에 유용하다.

### 7.3 Go To Line

- 특정 줄 번호로 즉시 이동
- 로그/설정 파일 확인 시 필요하다.

### 7.4 Toolbar Customization

- 자주 쓰는 버튼을 노출하고, 덜 쓰는 기능은 숨길 수 있는 구조
- 최소한 표시/비표시 수준의 간단한 커스터마이징은 확장 가능하게 설계

### 7.5 Recent Files / Reopen Closed Tab

- 최근 파일 목록
- 방금 닫은 탭 다시 열기

### 7.6 Auto Save for Saved Files

- 파일 경로가 있는 문서도 옵션에 따라 자동 저장 가능
- 기본값은 수동 저장으로 시작하고, 이후 설정으로 확장 가능

### 7.7 Encoding and Line Ending Selection

- 파일 열기/저장 시 인코딩 지정
- LF/CRLF 변경 지원
- 텍스트 유틸리티 성격이 강해질수록 중요해진다.

### 7.8 Theme and Font Preferences

- 설정 화면에서 테마 선택이 가능해야 한다.
- 테마는 자유 색상 편집보다, 미리 정의된 preset 선택 방식을 우선한다.
- 각 테마 preset은 최소 다음 값을 포함한다.
- 편집 배경 색
- 기본 글자 색
- 줄번호 색
- 현재 줄 강조 색
- 선택 영역 색
- 상태 바/툴바 색 조합
- 기본 폰트 크기
- 필요 시 기본 폰트 종류
- 다크 테마를 기본 제공한다.
- 추후 라이트 테마 또는 추가 preset 테마 확장 가능
- 사용자는 설정에서 배경 이미지를 선택할 수 있어야 한다.
- 배경 이미지는 편집 가독성을 해치지 않도록 opacity 또는 dim 처리 옵션이 필요하다.
- 배경 이미지는 전체 창 배경 또는 편집 영역 배경에 적용할 수 있는 구조가 바람직하다.
- 배경 이미지 사용 시 텍스트 대비를 유지하기 위한 overlay 색상 또는 blur 처리가 가능하면 좋다.
- 테마와 배경 이미지는 문서별이 아닌 앱 전역 설정을 기본으로 한다. 필요 시 추후 문서별 설정으로 확장 가능해야 한다.

### 7.9 Large File Safe Mode

- 매우 큰 파일 열기 시 일부 고급 기능을 제한하여 안정성 유지
- 예: 실시간 하이라이트/복잡한 레이아웃 기능 비활성화

### 7.10 Large File Progressive Loading

- 일정 크기 이상의 파일은 `Large File Mode`로 열 수 있어야 한다.
- Large File Mode에서는 파일 전체를 한 번에 메모리로 올리지 않고, 초기 표시 영역 중심으로 일부만 먼저 로드한다.
- 이후 스크롤 위치를 기준으로 필요한 구간을 점진적으로 읽어와 캐시해야 한다.
- 캐시는 현재 화면 주변 구간을 우선 유지하고, 멀어진 구간은 해제 가능해야 한다.
- 사용자는 파일이 완전히 로드되지 않은 상태임을 인지할 수 있어야 한다.
- 상태 바 또는 표시 배너에 `Large File Mode` 상태를 노출한다.
- 초기 오픈 시 최소 다음 동작이 필요하다.
- 파일 크기 확인
- 텍스트 파일 여부 확인
- 인코딩 추정
- 초기 청크 로드
- 백그라운드 추가 로드 또는 on-demand 로드
- Large File Mode에서는 일부 기능을 제한할 수 있다.
- 예: 전체 문서 replace, 전체 문서 통계, 복잡한 검색 결과 하이라이트, 실시간 줄 레이아웃 캐시
- 편집 가능 여부는 별도 정책으로 정의한다.
- 초기 버전에서는 대용량 파일을 읽기 전용으로 여는 방안이 더 현실적이다.
- 이후 버전에서 전체 편집 지원이 필요하면 별도의 버퍼/인덱싱 구조를 설계해야 한다.

## 10. UX Rules

- 저장되지 않은 탭을 닫을 때는 `Save / Don't Save / Cancel` 흐름 제공
- 임시 문서 복원 시 “복구된 문서”임을 사용자에게 명확히 알림
- 저장 실패, 권한 부족, 인코딩 실패는 조용히 무시하지 않고 알림 제공
- 비어 있는 신규 탭은 불필요한 경고 없이 닫을 수 있어야 한다.
- 검색 바는 편집 흐름을 방해하지 않는 방식으로 표시
- 대용량 파일이 제한 모드로 열리면, 제한되는 기능을 사용자에게 명확히 알려야 한다.
- 아직 로드되지 않은 구간이 있는 경우, 스크롤 중 빈 화면이나 점프 현상이 최소화되어야 한다.

## 11. Suggested MVP Definition

MVP에는 아래 범위를 포함한다.

- 상단 툴바(새 문서, 열기, 저장, 실행 취소/다시 실행, 폰트, 글자 크기, 검색)
- 탭 기반 문서 편집
- 검정 배경 중심의 편집 UI
- 줄번호 표시
- 소프트 랩
- 기본 텍스트 입력/복사/붙여넣기/실행 취소
- Tab 및 Shift+Tab 들여쓰기
- 새 문서 / 열기 / 저장 / 다른 이름으로 저장
- text 파일 드래그 앤 드롭 열기
- 인코딩 / 줄바꿈 형식 / 파일 타입 기본 감지
- 저장되지 않은 신규 문서 임시 저장 및 세션 복원
- 현재 문서 검색
- 상태 바의 줄/열/저장 상태 표시

## 12. Suggested Post-MVP Order

1. 최근 파일 / 닫은 탭 복원
2. 치환 기능
3. 설정 화면(폰트, 탭 정책, 줄바꿈 옵션)
4. 인코딩 및 줄바꿈 형식 제어
5. 대용량 파일 최적화
6. 읽기 전용/외부 변경 충돌 처리 강화
7. 대용량 파일 점진 로딩 및 캐시 고도화

## 13. Open Decisions

구현 전에 아래 항목은 확정이 필요하다.

- 에디터 엔진 선택
- SwiftUI 기본 `TextEditor` 확장인지
- AppKit `NSTextView` 브리징인지
- 탭 문자를 실제 `\t`로 저장할지, 공백으로 치환할지
- 자동 저장 기본 정책을 신규 임시 문서에만 적용할지, 저장된 문서까지 확장할지
- 세션 복원 기본값을 항상 켤지, 사용자 설정으로 둘지
- 검색/치환 UI를 인라인 바 형태로 할지, 별도 패널로 할지
- 툴바를 고정형으로 둘지, macOS window toolbar 스타일에 맞출지
- 파일 형식 감지 결과를 어디까지 UI에 노출할지
- 배경 이미지를 편집 영역에 직접 깔지, 창 전체 배경에만 적용할지
- 대용량 파일 기준 크기를 얼마로 둘지
- Large File Mode에서 편집을 허용할지, 초기에는 읽기 전용으로 제한할지
- 줄번호를 대용량 파일에서 정확 계산할지, 지연 계산/근사 계산할지

## 14. Recommended Technical Direction

현재 요구사항 기준으로는 SwiftUI 단독보다 AppKit의 `NSTextView` 기반 편집 엔진을 래핑하는 방향이 더 안전하다.

이유:

- 줄번호 구현이 수월하다.
- 텍스트 선택, 탭 처리, undo/redo 등 기본 편집 기능이 성숙하다.
- 대용량 텍스트 대응이 상대적으로 유리하다.
- macOS 네이티브 편집 경험에 가깝게 만들 수 있다.

단, 매우 큰 파일의 점진 로딩까지 진지하게 지원하려면 단순 `NSTextView`만으로는 한계가 있을 수 있다.

이 경우 방향은 두 가지다.

- 일반 파일: `NSTextView` 기반 일반 편집 모드
- 대용량 파일: 별도 데이터 소스/청크 캐시 기반 읽기 중심 모드

즉, 모든 파일을 하나의 편집 엔진으로 처리하려 하기보다, 파일 크기에 따라 모드를 분기하는 구조가 현실적이다.

즉, UI 셸은 SwiftUI로 두고, 핵심 편집 영역은 AppKit 브리징하는 구조를 권장한다.
