1. 표시 옵션 후속으로 `공백/탭 표시` 추가
2. 현재 줄 강조 on/off 토글 추가

주의할 점:

- `LLM Chat`, `Simple Terminal`, 고급 AI 액션은 현재 코드에 pane placeholder만 있고 우선순위는 뒤로 미룬다.
- 최근 이슈였던 "탭 복귀 시 텍스트가 안 보이는 문제"는 `CodeEditorContainerView`의 documentView 크기와 scroll bounds 보정으로 해결한 상태다.
- 최근 툴바는 텍스트 나열형에서 compact group UI로 바뀌었으므로, 다음 세션에서 버튼/메뉴 배치 미세조정은 이 구조를 기준으로 이어가면 된다.

## 1. Goal

이 문서는 `REQUIREMENTS.md`를 바탕으로 Text Viewer Editor를 구현 가능한 순서로 정리한 개발용 문서다.

목표는 다음과 같다.

- 의존성이 큰 기능부터 순서대로 구현한다.
- 각 단계가 끝날 때마다 실제로 실행 가능한 상태를 만든다.
- MVP 범위를 유지하고, 복잡도가 급격히 올라가는 기능은 제한적으로 구현한다.

본 문서는 특히 아래 기능을 중심으로 한다.

- 좌측 패널
- 파일 탐색기
- 검색 패널
- Text 창
- 미니맵
- 분할 레이아웃
- LLM 채팅
- 문서 내 AI 액션
- Simple Terminal
- 탭, 상태바, 세션 복원

MVP에 포함할 기능:

- 프로젝트 또는 작업 디렉터리 열기
- 좌측 패널
- 좌측 패널 내 세로 탭 전환
- 파일 탐색기 트리
- 검색 패널
- Text 창
- 탭 기반 파일 열기
- 상태바
- 우측 미니맵
- 최대 4분할 레이아웃
- 창 유형 전환: `Text`, `LLM Chat`, `Simple Terminal`
- 문서 내 AI 액션: 요약, 오타 보정, 이어쓰기, 톤 변환

- Command Palette
- 다중 커서
- 정교한 코드 에디터 수준 미니맵 렌더링

## 3. Architecture Overview

앱은 다음 세 영역으로 구성한다.

- 좌측 보조 패널
- 중앙 작업 영역
- 하단 상태바

중앙 작업 영역은 하나 이상의 pane으로 구성되며, 각 pane은 다음 중 하나의 창 유형을 가진다.

- `Text`
- `LLM Chat`
- `Simple Terminal`

각 pane은 독립적인 탭 상태를 가질 수 있다. 분할 레이아웃은 자유 배치가 아니라 preset 기반으로 관리한다.

권장 상태 모델:

- `WorkspaceState`
- `SidebarState`
- `LayoutState`
- `PaneState`
- `TabState`
- `DocumentState`
- `SearchState`
- `ChatSessionState`
- `TerminalSessionState`

핵심 원칙:

- pane, tab, document는 별도 ID로 관리한다.
- 현재 활성 pane과 활성 tab은 전역 상태에서 분리 관리한다.
- LLM과 Terminal은 UI는 pane에 속하지만 실행 상태는 별도 session 객체로 관리한다.
- 미니맵, 상태바, 검색 패널은 모두 현재 활성 `Text` 문서를 기준으로 동작한다.

## 4. Recommended Implementation Order

권장 구현 순서는 다음과 같다.

1. 앱 셸 및 전역 상태 구조
2. 프로젝트 열기와 파일 시스템 읽기
3. Text 창과 기본 문서 편집
4. 좌측 패널, 파일 탐색기, 검색 패널
5. 탭과 상태바
6. 분할 레이아웃 시스템
7. 미니맵
8. 문서 내 AI 액션
9. LLM 채팅 창
10. Simple Terminal
11. 세션 복원
12. 예외 처리 및 성능 정리

이 순서를 따르는 이유:

- `Text` 편집이 중심 기능이므로 가장 먼저 안정화해야 한다.
- 파일 탐색기와 검색이 있어야 실제 편집 흐름을 검증할 수 있다.
- 분할은 pane 상태 구조가 먼저 정리되어야 안전하다.
- LLM과 Terminal은 활성 문서, 프로젝트 경로, pane 상태에 의존한다.
- 세션 복원은 상태 구조가 굳은 뒤 구현해야 재작업이 적다.

## 5. Implementation Phases

## 5.1 Phase 1: App Shell and Global State

목표:

- 앱 기본 레이아웃 구성
- 상태 모델과 ID 체계 정의
- 활성 pane, 활성 tab, 현재 프로젝트 상태 관리

구현 항목:

- 메인 화면 레이아웃
- 좌측 패널 placeholder
- 중앙 작업 영역 placeholder
- 하단 상태바 placeholder
- pane, tab, document 모델 정의
- 레이아웃 preset enum 정의

완료 기준:

- 앱이 기본 레이아웃으로 실행된다.
- 빈 상태에서도 pane 타입 전환이 가능하다.
- 전역 상태만으로 활성 pane과 분할 레이아웃을 변경할 수 있다.

## 5.2 Phase 2: Project Open and File System Access

목표:

- 프로젝트 또는 작업 디렉터리를 열고 파일 목록을 읽을 수 있도록 한다.

구현 항목:

- 작업 디렉터리 선택
- 루트 경로 상태 저장
- 디렉터리 목록 읽기
- 파일 메타데이터 최소 모델 정의

주의 사항:

- 대용량 디렉터리를 고려해 lazy loading 구조로 설계한다.
- 숨김 파일 표시 여부는 MVP에서 고정값으로 시작해도 된다.
- 심볼릭 링크 처리 정책을 명확히 정한다.

완료 기준:

- 사용자가 작업 디렉터리를 선택할 수 있다.
- 루트 폴더의 파일/폴더 목록을 읽을 수 있다.

## 5.3 Phase 3: Text Window and Document Editing

목표:

- 파일을 열고 편집하고 저장하는 핵심 편집 흐름 구현

구현 항목:

- `Text` pane UI
- 파일 열기
- 문서 내용 로딩
- 편집
- 저장
- 수정 상태 표시
- 줄번호 표시
- 기본 soft wrap

설계 원칙:

- MVP에서는 UTF-8 중심으로 시작한다.
- dirty state는 문서 단위로 관리한다.
- 동일 파일을 다시 열 때 새 탭으로 열지, 기존 탭을 활성화할지 정책을 정한다.

완료 기준:

- 파일을 열어 수정 후 저장할 수 있다.
- 수정 여부가 UI에 반영된다.
- 줄번호와 기본 편집 동작이 안정적으로 작동한다.

현재 상태 메모:

- 완료
- 추가로 draft 세션 복원, 파일 형식 추정 일부, 외부 파일 변경 감지까지 연결됨

## 5.4 Phase 4: Left Sidebar, File Explorer, Search

목표:

- 좌측 패널을 추가하고 세로 탭으로 `파일 탐색기`와 `검색`을 전환할 수 있도록 한다.

구현 항목:

- 좌측 패널 프레임
- 세로 탭 UI
- 파일 탐색기 트리
- 검색 패널
- 좌측 패널 표시/숨김

파일 탐색기 범위:

- 트리 구조 표시
- 폴더 펼치기/접기
- 파일 선택 시 열기
- 현재 파일 하이라이트

검색 범위:

- 현재 문서 검색
- 검색 결과 목록
- 결과 선택 시 해당 위치 이동
- 본문 내 결과 강조

추가 판단:

- 프로젝트 전체 검색은 MVP 후반에 붙여도 된다.
- 검색은 새 창이 아니라 좌측 패널 탭으로 유지한다.

완료 기준:

- 좌측 패널에서 파일 탐색기와 검색을 전환할 수 있다.
- 파일 탐색기에서 파일을 열 수 있다.
- 검색 결과를 클릭해 본문 위치로 이동할 수 있다.

현재 상태 메모:

- 완료
- 검색 결과 본문 하이라이트, `Match Case`, 검색 상태 복원, `Enter`/`Shift+Enter` 이동까지 반영됨

## 5.5 Phase 5: Tabs and Status Bar

목표:

- 실제 편집기 사용성을 높이는 기본 보조 UI를 완성한다.

구현 항목:

- pane별 탭 바
- 다중 탭 전환
- 탭 닫기
- 수정됨 표시
- 상태바 정보 표시

상태바 최소 항목:

- 파일명
- 저장 상태
- 현재 줄
- 현재 열
- 현재 pane 유형

완료 기준:

- 하나의 `Text` pane에서 여러 파일을 탭으로 전환할 수 있다.
- 상태바가 현재 편집 상태를 정확히 반영한다.

현재 상태 메모:

- 완료
- 상태바에는 파일명, 저장 상태, 줄/열, 검색 상태, 문서 타입/인코딩 일부가 반영됨

## 5.6 Phase 6: Split Layout System

목표:

- 최대 4개 pane까지의 preset 기반 화면 분할 구현

구현 항목:

- 분할 preset 모델
- 레이아웃 선택 UI
- pane 생성 및 제거
- pane별 창 유형 선택

지원 레이아웃:

- 가로 2분할
- 가로 3분할
- 세로 2분할
- 세로 3분할
- 2x2 격자형
- 상단 2 / 하단 1
- 상단 1 / 하단 2

설계 원칙:

- 레이아웃 변경 시 기존 pane 상태를 최대한 유지한다.
- 각 pane은 독립적인 활성 탭과 창 유형을 가진다.

완료 기준:

- 사용자가 preset을 선택하면 즉시 레이아웃이 변경된다.
- 각 pane에 `Text`, `LLM Chat`, `Simple Terminal`을 독립적으로 배치할 수 있다.

## 5.7 Phase 7: Minimap

목표:

- `Text` pane 우측에 문서 위치 탐색용 미니맵 제공

구현 항목:

- 미니맵 렌더링 영역
- 현재 viewport 표시
- 본문 스크롤과 동기화
- 클릭 이동
- 미니맵 표시/숨김

제한 범위:

- MVP에서는 정교한 축소 렌더링보다 위치 탐색 기능에 집중한다.
- 긴 문서에서 성능이 급격히 떨어지지 않는 구조여야 한다.

완료 기준:

- 본문 스크롤과 미니맵이 동기화된다.
- 사용자가 미니맵을 클릭해 문서 위치를 이동할 수 있다.

## 5.8 Phase 8: Inline AI Actions

목표:

- 별도 채팅 창 없이 툴바, 버튼, 메뉴에서 AI 기능 실행

구현 항목:

- AI 액션 진입점 UI
- 현재 선택 텍스트 또는 문서 전달
- 요청 실행
- 응답 표시
- 적용 또는 취소 흐름

지원 기능:

- 요약
- 오타 보정
- 이어쓰기
- 톤 변환

기술 방향:

- macOS 기본 제공 LLM 사용
- 선택 영역이 있으면 우선 사용한다.
- 결과는 바로 반영하지 않고, 사용자가 검토 후 적용하는 흐름을 우선한다.

완료 기준:

- `Text` pane에서 AI 액션을 직접 실행할 수 있다.

완료 기준:

- 실행 중 취소가 가능하다.

## 6. Technical Notes

### 6.1 Text System

- 편집기 코어는 안정성과 성능을 우선한다.
- 줄번호, soft wrap, 검색 highlight, 미니맵 동기화를 고려한 구조가 필요하다.
- 나중에 문서 형식 감지, 치환, 고급 편집 기능을 붙일 수 있도록 모델을 분리한다.

### 6.2 Sidebar System

- 좌측 패널은 하나의 컨테이너로 유지한다.
- 기능은 세로 탭으로 전환한다.
- 추후 `Outline`, `Recent`, `Bookmarks`를 같은 구조에 쉽게 추가할 수 있어야 한다.

### 6.3 Split System

- 자유 분할보다 preset 기반 구조가 구현과 복원 모두에 유리하다.
- 분할 구조는 저장 및 복원이 쉬운 직렬화 가능한 모델로 관리한다.

### 6.4 AI Integration

- 기본 방향은 macOS 기본 제공 LLM 사용이다.
- 문서 내 액션과 채팅 창은 공통 LLM 서비스 레이어를 공유하는 편이 좋다.
- 프롬프트 생성, 응답 처리, 오류 처리를 UI와 분리한다.

### 6.5 Simple Terminal

- `Simple Terminal`은 터미널 에뮬레이터가 아니라 명령 실행 패널이다.
- 비대화형 작업을 우선 목표로 한다.
- 작업 디렉터리와 환경 정보는 항상 명확히 표시한다.

## 7. Risks and Scope Control

복잡도가 빠르게 증가하는 영역:

- 미니맵 렌더링 정밀도
- 프로젝트 전체 검색 성능
- 분할 상태 보존
- 장문 문서 기반 LLM 처리
- Terminal의 interactive 요구 대응

범위 통제 원칙:

- 각 기능은 먼저 동작하는 최소 버전으로 구현한다.
- `Simple Terminal`은 제한적 콘솔이라는 점을 유지한다.
- AI 결과는 자동 적용보다 검토 후 적용을 기본으로 한다.
- 분할은 자유 조절보다 preset 기반을 유지한다.

- gutter의 세로 좌표계를 본문 `NSScrollView` 오프셋과 동기화
- 기존 편집기 코드는 보조 경로로 남기고 새 편집기를 기본 연결

다음 세션 첫 우선순위:

1. 빈 줄 / soft wrap / 긴 문서에서 줄번호와 현재 줄 계산 추가 검증
